Client–Server Architecture: Tiers (1, 2, 3, N)

Quick Overview

A tier is a logical layer in a client–server system. We can combine or separate the presentation, business logic, and data into different tiers to balance simplicity, scalability, and security.

Rule of thumb:
More tiers = more flexibility and scalability, but also more complexity.

Example: Classic 3-Tier Architecture

In a 3-tier system, each layer has a clear role:

Presentation Tier (Client) Browser, mobile app, desktop UI
↓
Application Tier (Server) Business logic, APIs, validation
↓
Data Tier (Database) Relational DB, files, storage

Tier Comparison

Tier Type Layers Where is Logic? Main Advantages Main Disadvantages
1-Tier One program with UI + logic + data All inside the same executable Very simple; no network; fast for a single user Not scalable; hard to share data; hard to maintain
2-Tier Client + Database server Mainly in the client, some in DB Simpler than 3-tier; good performance; central DB Many clients to update; DB more exposed; limited scaling
3-Tier Presentation, Application, Data Application server (web/API layer) Separation of concerns; better security; easier scaling More components; more complexity to manage
N-Tier Client, gateways, services, DB, cache, etc. Split across multiple services Highly scalable and flexible; fault isolation Very complex; needs strong DevOps and monitoring
