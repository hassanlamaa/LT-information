1. Principles

Definition Client/Server is a distributed model where clients request services and servers provide them. Clients handle presentation/UI, while servers centralize data and services.

1.1 Communication (Request/Response)

  • Client sends a request → Server processes → Server returns a response.
  • Protocol: agreed rules (HTTP, FTP, SMTP, etc.).
  • IP Address: identifies hosts; Port: identifies service endpoints.
  • Scheduling: servers prioritize and limit requests to protect resources.
Scenario: Online Banking
  1. Browser (client) sends login to a web server.
  2. Web server queries database servers (may chain to others).
  3. Application server applies business logic.
  4. Web server responds with account data.

1.2 Advantages & Disadvantages

AdvantagesDisadvantages
Centralized management (backup, security, admin)Server cost and complexity
Scalability & flexible resource growthSingle point of failure (server outages)
Enhanced data sharing & access controlPotential congestion with many clients

2. Communication & Protocols

Ports (Well-known examples)

  • HTTP 80, HTTPS 443, FTP 21/20, SMTP 25, POP3 110, IMAP 143
  • Registered: 1024–49151, Dynamic: 49152–65535

TCP

  • Reliable, connection-oriented, in-order delivery.
  • Use cases: Web, Email, File transfer (HTTP, HTTPS, FTP, SMTP…)
  • Header ~20 bytes + options; flow control, congestion control.

UDP

  • Unreliable, connectionless, fast; per-datagram delivery.
  • Use cases: DNS, VoIP, streaming, SNMP, DHCP.
  • Header 8 bytes; apps handle ordering and reliability if needed.

3. Server Types & Roles

File Server

Centralized file storage and retrieval.

Database Server

DBMS services; clients send SQL; server executes queries.

Transaction Server

Groups SQL into atomic transactions (OLTP).

Groupware Server

Collaboration tools (calendars, shared docs, meetings).

Web Server

Serves HTTP static content.

Application Server

Generates dynamic content; business logic layer.

4. Architectures (2-Tier, 3-Tier, N-Tier)

2-Tier

Client (UI + some logic) → Database server (data + logic). Simple, high performance locally, but limited scalability.

3-Tier

Client (presentation) → Application server (business logic) → Database server (data). Better scalability and security.

N-Tier

Multiple middle-tier components (microservices/components). Improved maintainability and load distribution.

6. HTTP Protocol (Case Study)

Key Ideas

  • Application-level protocol over TCP (default port 80).
  • Connectionless and stateless; media-type independent via MIME.
  • Methods: GET, HEAD, POST, PUT, DELETE, OPTIONS, TRACE, CONNECT.
  • Status codes: 1xx info, 2xx success, 3xx redirection, 4xx client error, 5xx server error.
